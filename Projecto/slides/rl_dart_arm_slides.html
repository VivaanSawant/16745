Reinforcement Learning for the Dart-Throwing Arm

Using RL to learn how the arm should move to maximize score under uncertainty

The problem

  • We have a robot arm that throws a dart at a board.
  • Small changes (release timing, torque, wind) → different landing.
  • We don’t want one perfect throw — we want a policy that gets high expected score even with noise.

Why use RL to “figure out the arm”?

  • Arm dynamics + release + flight are complex and stochastic.
  • Closed-form “do this torque at this time” is hard to get right.
  • RL learns a mapping: state → action by trial and error in simulation.
  • The policy is our way of encoding “how the arm should work” for this task.

High-level RL loop

State → Policy (arm) → Action → Physics sim → Landing → Reward

  • State: joint angles, joint velocities, (optional) wind estimate.
  • Action: joint torques (e.g. wrist), release timing.
  • Sim: our 3D projectile + dartboard (what we already built).
  • Reward: dartboard score (and maybe smoothness / stability penalties).

State space

  • Arm: joint angles θ, joint angular velocities θ̇.
  • Optional: estimated wind, or dart orientation at release.
  • From state we can compute (or estimate) release position & velocity → then run our physics to get landing.

Action space

  • Continuous: e.g. wrist torque, other joint torques over a short throw phase.
  • Release timing: when to release the dart (discrete event in an otherwise continuous control).
  • We train a policy (e.g. neural network) that outputs these actions given the current state.

Reward

  • Primary: dartboard score at landing (from our score_at(x, y)).
  • Optional: penalize unstable release, large torques, or variance across rollouts.
  • We can use expected score (Monte Carlo over many noisy rollouts) so the policy optimizes for robustness.

How we “figure out how the arm works”

  • We don’t hand-design torques or timings.
  • We define state, action, reward and let RL (e.g. SAC, TD3, PPO) learn a policy.
  • The learned policy = “given this arm state (and env), do this action.”
  • So the arm’s behavior is discovered by the algorithm using our physics + dartboard as the environment.

Training pipeline

  1. Build an RL environment: step(state, action) → run arm dynamics to release, then our projectile + dartboard to get landing and score.
  2. Use a standard off-the-shelf RL algorithm (e.g. Stable-Baselines3) to train the policy.
  3. Optionally add curriculum: start with little noise, then increase wind/release variance.
  4. Evaluate with Monte Carlo: many rollouts per policy → distribution of landings and expected score.

What we already have

  • 3D projectile physics (drag, wind).
  • Dartboard scoring and “hit any number.”
  • Release variance and Monte Carlo landings.
  • Simple state estimator (arm state → release pos/vel).
  • Next: wrap this in an RL env (state/action/reward) and train a policy.

Summary

  • RL learns a policy that maps arm (and env) state to torques and release timing.
  • Our physics simulator provides the transition: action → release → trajectory → landing → score.
  • By maximizing expected score under noise, we get a robust “recipe” for how the arm should move — that’s what we mean by figuring out how the arm works with RL.
